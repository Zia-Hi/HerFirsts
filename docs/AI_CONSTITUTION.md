# AI CONSTITUTION

Version 1.0

This document defines the development rules for every AI working on Her Firsts.

Failure to follow these rules means the task is incorrect.

---

# Rule 1

Read every document inside /docs before writing any code.

Never assume project requirements.

---

# Rule 2

This project is a GAME.

Never design a website.

Never use SaaS layouts.

Never create dashboards.

Never create landing pages similar to startup websites.

---

# Rule 3

One prompt = One task.

Never build multiple systems simultaneously.

Stop after completing the requested milestone.

Wait for review.

---

# Rule 4

Never modify unrelated files.

Only edit files required for the current task.

Always list every modified file.

---

# Rule 5

Generate production-ready code.

Never leave TODO comments.

Never write placeholder logic.

Never generate pseudo-code.

Everything should run.

---

# Rule 6

Do not redesign previous work unless explicitly instructed.

Respect existing architecture.

---

# Rule 7

Always follow:

PROJECT_CONTEXT.md

GDD.md

ART_DIRECTION.md

GAMEPLAY_PRINCIPLES.md

If conflicts exist:

Gameplay Principles

↓

Art Direction

↓

GDD

↓

Project Context

---

# Rule 8

Code quality is more important than speed.

Readable code.

Reusable components.

No duplicated logic.

---

# Rule 9

Every animation must have purpose.

Never animate simply because animation looks cool.

Every animation should support storytelling.

---

# Rule 10

Every interaction must feel physical.

Objects have weight.

Objects have sound.

Objects have intention.

---

# Rule 11

The player should never feel lost.

Do not use traditional tutorial UI.

Instead:

Lighting

Composition

Animation

Sound

Environmental storytelling

should guide the player.

---

# Rule 12

Never generate generic AI aesthetics.

Avoid:

Rounded cards

Floating panels

Hero banners

Glassmorphism

Neon

Large gradients

Oversized shadows

Modern startup UI

---

# Rule 13

The world is the interface.

Buttons are the last resort.

Whenever possible:

Open a door instead of clicking Start.

Use a notebook instead of a menu.

Use a toolbox instead of an inventory window.

---

# Rule 14

After every completed task, output:

## Files Modified

...

## Files Created

...

## What Changed

...

## How to Test

...

Then STOP.

Wait for the next instruction.
